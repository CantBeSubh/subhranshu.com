# Metrics Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire up Google Analytics (GA4), Microsoft Clarity, Vercel Analytics, and Vercel Speed Insights, gated to production builds only, via a new `Metrics` component mounted once in `src/app/layout.tsx`.

**Architecture:** A `src/components/metrics/` module with two leaf provider components (`google-analytics.tsx` wrapping `@next/third-parties/google`, `clarity.tsx` with a hand-rolled `next/script` snippet ported from the old site) composed by an `index.tsx` orchestrator that also renders Vercel's `Analytics`/`SpeedInsights` components and returns `null` outside `NODE_ENV === "production"`.

**Tech Stack:** Next.js (App Router), TypeScript, `@next/third-parties`, `@vercel/analytics`, `@vercel/speed-insights`, `bun` as package manager/runner.

## Global Constraints

- Use `bun` for all installs and verification commands (`bun add`, `bun run build`, `bun run lint`) — this repo has `bun.lock`, not `package-lock.json`.
- `NEXT_PUBLIC_GOOGLE_ANALYTICS` and `NEXT_PUBLIC_MICROSOFT_CLARITY` already exist in `.env.local` — do not add or modify env vars.
- New files live under `src/components/metrics/`, not `src/app/` — this project keeps non-route code in `components/`.
- All four providers must be gated behind `process.env.NODE_ENV !== "production"` returning `null` — none should fire in `bun run dev`.
- Match existing code style: double-quoted strings, semicolons, default exports for single-purpose components (see `src/components/section/contact-submit-button.tsx`).
- Do not commit `docs/superpowers/**` files — write them to disk only.
- No test framework exists in this repo — verification is `bun run lint`, `bun run build`, and manual dev/production checks.
- After all tasks are complete, the implementing agent must hand verification to the user rather than self-declaring success — walk them through the dev-mode and production-mode manual checks in Task 3 and let them confirm the result.

- [ ] **Step 1: Write `src/components/metrics/index.tsx`**

```tsx
import { Analytics as VercelAnalytics } from "@vercel/analytics/next";
import { SpeedInsights } from "@vercel/speed-insights/next";

import GoogleAnalytics from "./google-analytics";
import MicrosoftClarity from "./clarity";

const Metrics = () => {
  if (process.env.NODE_ENV !== "production") return null;

  return (
    <>
      <GoogleAnalytics />
      <MicrosoftClarity />
      <VercelAnalytics />
      <SpeedInsights />
    </>
  );
};

export default Metrics;
```

## Self-Review Notes

- **Spec coverage:** Task 1 covers spec §2–3 (Google Analytics via `@next/third-parties`, Clarity ported verbatim) and the dependency install. Task 2 covers spec §4 (orchestrator with uniform production gating across all four providers). Task 3 covers spec §5 (layout integration) and the Verification section (build/lint plus manual dev/production checks), including the user's explicit requirement that final verification be done by the user, not self-declared.
- **Type consistency:** `GoogleAnalytics` (default export, no props) in Task 1 Step 2 matches the bare `<GoogleAnalytics />` usage in Task 2 Step 1. `MicrosoftClarity` (default export, no props) in Task 1 Step 3 matches `<MicrosoftClarity />` in Task 2 Step 1. `Metrics` (default export, no props) in Task 2 Step 1 matches the bare `<Metrics />` usage in Task 3 Step 2.
- **Naming collision avoided:** `@next/third-parties/google`'s exported `GoogleAnalytics` is aliased to `NextGoogleAnalytics` inside `google-analytics.tsx` so the file's own default-exported `GoogleAnalytics` doesn't shadow it. Similarly `@vercel/analytics/next`'s `Analytics` is aliased to `VercelAnalytics` in `index.tsx` to keep the identifier unambiguous next to Google/Clarity/Vercel's own naming.
